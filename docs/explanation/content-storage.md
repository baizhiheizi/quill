# Content storage

> **30-second summary:** Published article content lives in the Quill application itself — the formatted article body in **ActionText**, a `legacy_markdown_content` fallback for pre-migration rows, the snapshot history in **`ArticleSnapshot`** (PostgreSQL JSON), and per-article images, posters, and covers in **ActiveStorage**. Quill does **not** mirror content to any external permanence network. The application is the single source of truth.

## Why this page exists

Earlier versions of Quill uploaded published content to [Arweave](https://www.arweave.org/) and offered a [Mirror.xyz](https://mirror.xyz/) import flow. Both were removed in #1582; the change is captured in [`openspec/specs/content-storage/spec.md`](../../openspec/specs/content-storage/spec.md) as the formal `content-storage` capability. A future contributor should be able to read this page and understand what guarantees that capability makes — and what is explicitly out of scope.

## What is stored where

| Asset | Lives in | Notes |
|-------|----------|-------|
| Article body (current, formatted) | `action_text_rich_texts` (ActionText) | Set up via `has_rich_text :content` in `RichTextContent`; the canonical store for migrated articles |
| Article body (legacy Markdown) | `articles.legacy_markdown_content` (`text`) | Fallback for articles written before the ActionText migration; `migrated_content?` switches readers between the two |
| Article metadata (`title`, `intro`, `price`, `asset_id`, `state`, `*_revenue_ratio`, …) | `articles` table (PostgreSQL) | The `Article` model itself |
| Content history | `article_snapshots.raw` (PostgreSQL JSON) | One row per save that touches `title`, `intro`, `content`, or `published_at`; `store_accessor :raw` exposes `title`, `intro`, `content`, `digest` |
| Inline images referenced from the body | `active_storage_blobs` (`has_many_attached :images`) | Referenced from the ActionText Trix content |
| Per-article social-share poster | `active_storage_blobs` (`has_one_attached :poster`) | Generated by `Articles::GeneratePosterJob` on publish |
| Per-article cover image | `active_storage_blobs` (`has_one_attached :cover`) | Optional, set by the author |

`Article#content` (the public reader) goes through `RichTextContent#content_as_html`, which renders from ActionText when `migrated_content?` is true and falls back to `legacy_markdown_content` (rendered through `MarkdownRenderService`) otherwise. `Article#content=` writes through ActionText and the `after_save` hook triggers `generate_snapshot` if `should_generate_snapshot?` returns true.

## What the publishing flow does — and does not

When an author runs `Article#publish!`:

1. The `Article` row transitions to the `published` state via AASM.
2. `notify_for_first_published` fires (see [Reference → Notifiers](../reference/notifiers.md#articlepublishednotifier)).
3. `Articles::GeneratePosterJob` enqueues a social-share poster render on the `low` queue.
4. `Articles::CreateWalletJob` provisions a Mixin wallet for the article (one-time).

That is the complete list. There is **no** enqueue for an external upload, **no** outbound HTTP call to a permanence network, and **no** background job that "mirrors" content elsewhere. The poster-render job writes a derived asset (the Open Graph image) back into ActiveStorage; it does not leave the application.

## What was removed

The Arweave integration shipped behind several touch points, all of which are now gone:

- `app/jobs/articles/arweave_upload_job.rb` and related Mixin-bot deliveries — deleted.
- `app/models/arweave_transaction.rb` and the `arweave_transactions` table — dropped in migration `db/migrate/20260611052000_drop_arweave_transactions.rb`.
- The `arweave` and `graphql-client` gems — removed from `Gemfile`.
- The author-side **"Web3 Proof of Publishing"** panel on the dashboard — removed.
- The Mirror.xyz import flow — removed.
- Admin moderation tooling for Arweave retries — removed.

The historical migration files (`db/migrate/20220810031609_create_arweave_transactions.rb` and its follow-ups) are intentionally left in place per Rails conventions; they are inert on a fresh database.

## What did not change

- The publish flow, including the `published` AASM state and `Article#may_publish?` guard.
- `ArticleSnapshot` creation on every `save` — snapshotting continues to run for content history. The snapshot row (`article_snapshots.raw`) is the only durable record of historical content beyond the current ActionText / `legacy_markdown_content` body.
- The API and reader access path. `GET /api/articles/:uuid` still returns the body to authorized readers via `RichTextContent#content_as_html`; no client change is required.
- The `solid_queue` and `solid_cable` infrastructure — unrelated and untouched.

## Why a future "add storage X" change must respect the capability

The `content-storage` capability asserts that Quill stores published content in PostgreSQL and ActiveStorage and serves it to authorized readers without external lookups. A subsequent change that proposes to reintroduce an external mirror (Arweave, Filecoin, IPFS, …) must update [`openspec/specs/content-storage/spec.md`](../../openspec/specs/content-storage/spec.md) **first** — the spec is the source of truth, not this page. Without the spec change, the integration is a violation of the documented contract.
