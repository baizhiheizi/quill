## Coverage

`MixinNetworkUser` test file was an empty placeholder. Filled with:

- **Validations**: name / pin_token / private_key / uuid / session_id all required (presence).
- **UUID uniqueness** is a DB-level constraint, not a model-level validation (no `validates :uuid, uniqueness: true`). Tested via `assert_raises(ActiveRecord::RecordNotUnique)` on save.
- **Associations**: `belongs_to :owner` optional + polymorphic (Article); `has_many :snapshots / :transfers` (dependent: :nullify).
- **Scopes**: `:ready` / `:unready` (SQL-only — `where.not(pin: nil)` translates to `WHERE encrypted_pin IS NOT NULL`).
- **`default_name`**: `Settings.broker_name || "Quill"` — tested both branches.
- **`avatar`**: prefers `raw["avatar_url"]`, falls back to `User.default_avatar_url`. Empty-hash case covered; nil-raw case would raise `NoMethodError` (model bug, see Side-findings).
- **`mixin_api`**: builds `MixinBot::API.new(client_id:, client_secret: nil, session_id:, pin_token:, private_key:)` and memoizes. Tested via `MixinBot::API.define_singleton_method(:new, …)` to capture kwargs without instantiating the real gem.
- **`verify_pin`** delegates to `mixin_api.verify_pin(pin)`.
- **`initialize_pin!`**: short-circuits when pin present, otherwise calls `update_pin!`. Both branches tested.
- **`update_pin!`**: generates a 6-digit pin (`SecureRandom.random_number.to_s.split(".").last.first(6)`), calls `mixin_api.update_pin(old_pin:, pin:)`, raises on empty `data`, updates `pin` on success.
- **`sync_profile!`**: calls `mixin_api.me`, raises on empty data, updates `raw` on success.
- **`update_name`**: calls `mixin_api.update_me(full_name:)`, persists `name + raw` on data present.
- **`update_avatar`**: opens `app/assets/images/icon.png`, sends `full_name + avatar_base64`, updates `raw` on success, closes the file in `ensure` even when the API raises.
- **`setup_attributes`** (before_validation on :create): populates `uuid / name / pin_token / session_id / private_key` from `QuillBot.api.create_user(name || default_name)` response; raises when response has `error`.
- **`after_commit`**: enqueues `MixinNetworkUsers::InitializePinJob` + `MixinNetworkUsers::UpdateAvatarJob` on create; does NOT fire on update (verified with `assert_no_enqueued_jobs`).

## Gotchas (new this run)

- **`before_validation :setup_attributes, on: :create` fires on every `valid?` call, not just `save`.** Tests that only check validations need to stub `setup_attributes` to a no-op via `define_singleton_method`. The `build_wallet` helper in the test file does this; the global `setup` block also stubs `QuillBot.api.create_user` because the callback fires on every new record's validation.
- **`encrypts :pin` + no `active_record_encryption.primary_key` in test credentials → `pin` accessor raises.** Workaround: stub `update!` / `update` on the instance (captures args, returns true), set `encrypted_pin` directly via `update_column(:raw, …)`, stub `setup_attributes` in `build_wallet`. Adding encryption keys to `config/credentials/test.yml.enc` would unlock real round-trip tests but is a breaking change for any test depending on the unconfigured state.
- **`MixinBot::API.singleton_class.define_method(:new)` evaluates `self` as `MixinBot::API`, so `assert_equal` doesn't work inside the override.** Capture kwargs in a closure (array / hash) and assert after the call.
- **`MixinBot::API.define_singleton_method(:new, original_method)`** is the right way to restore — `remove_method` would leave the class without a `new`.
- **Avatar nil-raw raises** (`raw["avatar_url"]` on nil). Documented as a side-finding for the maintainer.

## Side-findings (for Maintainer)

- **`MixinNetworkUser#avatar` does `raw["avatar_url"]` without a nil-guard on `raw`.** A direct write of `raw: nil` raises `NoMethodError`. Worth a one-line `raw&.[]("avatar_url")&.presence || User.default_avatar_url`.
- **ActiveRecord encryption is unconfigured in test credentials.** Adding `active_record_encryption:` to `config/credentials/test.yml.enc` would unlock real round-trip tests for `MixinNetworkUser#pin` and any future `encrypts` usage. **Risk:** it could surface latent encryption bugs elsewhere — keep an eye on the broader test suite if enabled.
