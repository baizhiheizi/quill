## Helper gotchas (new this run)

- **`Order#setup_attributes` requires a real `Payment.amount`.** Bypass cleanly with `Order.new(...).save(validate: false)` — `before_validation` callbacks DO NOT fire when `validate: false`. The order persists with whatever `buyer:`, `seller:`, `item:` were set in the constructor.
- **`Article` validations are heavyweight**: `validate_rich_text_content_presence?` (skipped only if `state == "drafted"`), `intro` presence, and `cannot_edit_frozen_attributes_once_published` (fires when `published_at` is present, even on new records — `asset_id_changed?` returns true). `Article.new(...).save(validate: false)` bypasses all.
- **`idx_orders_buyer_item_type_unique` is a DB-level unique constraint on `(order_type, buyer_id, item_type, item_id)`.** `save(validate: false)` skips the Ruby uniqueness validator but the DB still enforces it. To create multiple orders for the same buyer, give each a freshly-created Article (each `build_article` generates a unique UUID).
- **Fixture `Order.completed` baseline is 0**: `test/fixtures/orders.yml` has two unnamed orders (`one`, `two`) with `state: nil` → AASM initial = `paid`, NOT in `Order.completed`. `fixture_completed_orders_in_day_count` (a delta helper) is unnecessary because the baseline is genuinely zero — assert the absolute count instead.
- **`before_validation` does NOT fire under `save(validate: false)`** — confirmed by direct testing. Each test in this file uses `save(validate: false)` per-instance.

## Side-finding (for Maintainer)

`DailyStatistic#data_attributes[:paid_users_count]` uses `created_at: ...date.end_of_day` (open-start range, equivalent to `nil...date.end_of_day`). It counts **every buyer who's ever had a completed order up to and including the day**, not just within the day. Its sibling `new_payers_count` uses a bounded `date.beginning_of_day...date.end_of_day` range. They will diverge the first time any buyer has a completed order more than a day old. Flagged in PR description; not asserted as a test (would document divergent contracts as passing-by-coincidence).
