## Bug fixed (existing test)

The previous `process_job_test.rb` used `define_singleton_method(:find_by)` + `remove_method(:find_by)` for teardown. If the test raised before the `ensure` block fired, `MixinNetworkSnapshot.find_by` would be left uninstalled and break every subsequent test in the same process — same class-pollution pattern flagged in 2026-07-02's `UserAuthorization#has_safe?` notes. Swapped for `stub_class_method` (from `test/test_helper.rb`), which restores the original `UnboundMethod` from the class.

## Gotchas (new this run)

- **`define_singleton_method` + `remove_method` teardown is unsafe** — applies to *any* test file in this repo that stubs class methods this way. Generalize: prefer `stub_class_method` (in `JobTestCase`) when stubbing a class method for the duration of a test block. The class pollution also silently passes when the test body doesn't raise.
- **`MixinNetworkSnapshots::MonitorJob` reaches `AdminNotificationService` indirectly** — tests should not need to stub `AdminNotificationService` for the no-op path; the existing `delayed.count.zero?` short-circuit prevents the instantiation. Verified by stubbing `AdminNotificationService.new` and asserting it's NOT called when count is zero.

## Side-findings (for Maintainer)

- **`Bonus#table_name` mismatch persists** — `Bonus.create!` still raises `PG::UndefinedTable: relation "bonus" does not exist`. The model has no explicit `self.table_name = "bonuses"`, so Rails infers `"bonus"` from the class name, but `db/schema.rb` creates `"bonuses"`. One-line production fix: add `self.table_name = "bonuses"` to `app/models/bonus.rb`. Would unblock the entire `Bonus` AASM (`drafted → delivering → completed`) test coverage that has been in the backlog for 8+ runs.
- **`process_job_test.rb` and similar stubs in the codebase may have the same `remove_method` bug** — worth a grep pass to find any other `define_singleton_method` + `remove_method` pattern that could leave a class in a polluted state on test failure.
