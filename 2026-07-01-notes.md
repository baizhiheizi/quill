## Gotchas

- **Article `cannot_edit_frozen_attributes_once_published`** fires on the first save when `published_at` is set in the constructor (`asset_id_changed?` returns true). Workaround: create with `state: :drafted` (no `published_at`), then call `article.publish!` to transition via AASM. The `do_first_publish` after-callback sets `published_at = Time.current` cleanly.
- **`Article` requires `content`**: `has_rich_text :content` triggers `ensure_content_valid` (`title.present? && content.present?`) on the `publish` event. Set `article.content = "<p>test</p>"` before `save!` when authoring state is not :drafted.
- **`Noticed::Event.type` is an STI column.** Use a real notifier class name (`TransferProcessedNotifier`) in tests — fake names like `"SomeOtherNotifier"` raise `ActiveRecord::SubclassNotFound` on `.type` lookup.
- **`noticed_notifications` doesn't denormalise `record`.** To count notifications for a particular record: `Noticed::Notification.joins(:event).where(noticed_events: { record_type: "Tagging", record_id: id })`.
- **`Tagging#notify_subscribers` is on `after_create_commit`, but it's also triggered by an explicit `tagging.send(:notify_subscribers)` call.** The notifier serialises params (including `tagging`), so the tagging MUST be saved (have an id) before invoking `notify_subscribers` directly. Use `Tagging.create!` not `Tagging.new` in manual-invocation tests.
- **`User#blocked_user_ids_relation` is "users I have blocked"**, not "users who blocked me". `User.where.not(id: author.blocked_user_ids_relation)` = filter out users that the AUTHOR has blocked. Easy to read backwards in tests — see commit history `728a8a9` for the correction.
