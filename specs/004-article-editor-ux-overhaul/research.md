# Phase 0 Research: Article Editor Redesign — Modern, Unified Writing & Publishing Experience

All scope-defining questions were already resolved interactively during `/speckit-specify` and `/speckit-clarify` (recorded in `spec.md`'s `## Clarifications` and `## Assumptions`). No `NEEDS CLARIFICATION` markers remain. This document resolves the *technical approach* questions needed to plan confidently, grounded in direct inspection of the current implementation.

## 1. How should autosave be unified across content and settings, and across new/existing articles?

**Decision**: Eliminate the current two-tier save model (`update_content` for content, `update` for settings) and route **all** autosaves — content and settings alike — through the existing `PATCH/PUT /articles/:uuid` (`articles#update`) action, which already has the correct conditional strong-parameter logic (`update_article_params`) for published vs. drafted state. Remove the `update_content` action, its route (`put :update_content`), and its dedicated turbo_stream view. For a **brand-new, not-yet-persisted** article, the first autosave triggers a background `POST /articles` (`articles#create`); the response (JSON, not a full-page redirect) returns the new article's `uuid`, and the client swaps its autosave target to `PATCH /articles/:uuid` and updates the address bar via `history.replaceState` — no full page reload, no navigation-triggered loss of focus/cursor/scroll position.

**Rationale**:
- `articles#update` (`app/controllers/articles_controller.rb:63-70`) already wraps `CreateTagService.call` + `@article.update!` in a transaction and already has the exact conditional permitting (`update_article_params`, lines 125-157) the settings panel needs (e.g., price only permitted under certain conditions, revenue ratios frozen once published) — reusing it for autosave means zero duplicated business logic between "the manual settings save" (today) and "autosave" (this feature).
- `articles#update_content` (lines 72-76) is materially simpler and, per direct inspection, has a real bug: it looks up the article via `params[:article_uuid]`, but the route is declared `resources :articles, ..., param: :uuid` (`config/routes.rb:57`), meaning the member route param is `:uuid`, not `:article_uuid`. `params[:article_uuid]` will be `nil` on every request to this action, so `current_user.articles.find_by uuid: nil` returns `nil`, and the subsequent `@article.assign_attributes` raises `NoMethodError` on `nil` — i.e., **every content-only autosave request today fails server-side**, silently, from the client's point of view (it falls back to `localStorage` and retries, per `article_form_controller.js:372-414`, but never actually reaches the account). Consolidating onto `update` (which correctly uses `params[:uuid]` via `load_article`, line 159-162) fixes this bug (FR-023) as a side effect of the consolidation, rather than as a separate patch to code being deleted anyway.
- Consolidating removes the root cause of the "split save model" (audit finding #1 / FR-019): today, content changes are autosaved but settings changes are not, because they're two different code paths with two different reliability profiles. One endpoint, one save/status state, one set of retry/local-backup semantics.

**Current relevant code** (verified by reading):
- `app/controllers/articles_controller.rb:53-76` (`create`, `update`, `update_content`)
- `app/javascript/controllers/article_form_controller.js:372-414` (`autosave()`, using `@rails/request.js`'s `put()` against `autosaveUrlValue`)
- `config/routes.rb:57-61` (`resources :articles, ..., param: :uuid do put :update_content ... end`)
- `app/views/articles/_edit_form.html.erb:10` (`data-article-form-autosave-url-value="<%= article_update_content_path(article) %>"`)

**Alternatives considered**:
- *Keep two endpoints, just fix the `update_content` bug*: rejected — leaves the split-save-model root cause (FR-019) unaddressed; settings would still require a separate explicit "Save."
- *Introduce a brand-new unified `PATCH /articles/:uuid/autosave` endpoint, distinct from `update`*: rejected as unnecessary duplication — `update` already has every piece of conditional logic autosave needs; a second near-identical action would be the "parallel implementation" the project's rules explicitly discourage.

## 2. How does a brand-new article get created without a manual "Save Draft" click, without polluting the database with abandoned empty drafts?

**Decision**: Lazily create the `Article` record on the **first meaningful autosave** (the same 1-second-debounced trigger already used for content, extended to also fire on the first settings change) — not eagerly on `GET /articles/new` page load. "Meaningful" means the same threshold that already exists implicitly today: any non-empty title, intro, content, or a settings field changed from its pre-filled default. The `new.html.erb` page's Stimulus controller already carries a `newRecordValue` flag; extend its `autosave()`/new save method to `POST` to `articles_path` (via `@rails/request.js`'s `post()`, mirroring the existing `put()` usage) instead of `form.submit()`.

**Rationale**: This is functionally equivalent to today's behavior (a draft is only ever created once the author has actually typed something and it gets saved) but removes the requirement that the author click a button to trigger that save — it happens automatically, the same way Notion/Google Docs-style editors behave. It does not introduce a new class of "abandoned empty draft" risk beyond what already exists today (an author who clicks "Save Draft" once with minimal content already creates a persisted, mostly-empty draft; the existing "Delete draft" flow (`dashboard/deleted_articles_controller.rb`) already handles cleanup).

**Current relevant code**:
- `app/controllers/articles_controller.rb:45-61` (`new`, `create`)
- `app/javascript/controllers/article_form_controller.js:200-210` (`submit()` — currently does `form.submit()` for new records, a full-page POST)
- `app/views/articles/new.html.erb`, `app/views/articles/_edit_form.html.erb:16` (`data-article-form-new-record-value="false"` — hardcoded false here since this partial is edit-only; `new.html.erb` sets it `true`)

**Alternatives considered**:
- *Eagerly create an empty drafted `Article` the moment `GET /articles/new` is requested*: rejected — creates a persisted row for every visit to the "new article" page even if the author immediately navigates away, a regression in data hygiene versus today's explicit-save gate.
- *Keep the full-page `form.submit()` POST for the very first save, just remove the "Save Draft" button and trigger it automatically via JS*: rejected — still causes a full page navigation/reload on first meaningful keystroke, which is jarring and loses in-progress focus/cursor state; contradicts the "modern, pro" goal and the Focus Mode story's requirement (FR-026) that state survives seamlessly.

## 3. How does the system guarantee "last-writer-wins by edit recency," including across two tabs, without building real-time collaborative editing?

**Decision**: Add Rails' built-in optimistic locking to `Article` via a standard `lock_version` integer column (a normal, reversible migration: `add_column :articles, :lock_version, :integer, default: 0, null: false`). Every autosave `PATCH` includes the `lock_version` the client last saw; `ActiveRecord`'s native optimistic-locking check automatically raises `ActiveRecord::StaleObjectError` if another save has landed in between. `articles#update` rescues this alongside the existing `ActiveRecord::RecordInvalid` rescue and responds with a distinct "conflict" turbo_stream/status (409) that tells the client to reload the article's current state (a simple, safe "someone/something else just saved — here's the latest version" resolution) rather than silently overwriting it. Client-side, autosave requests are also serialized (never more than one in flight at a time; a newer edit queues behind the current request rather than firing concurrently), which eliminates the most common source of out-of-order writes without any server change.

**Rationale**: This is a solo-author writing tool, not a real-time collaborative document editor — the realistic "conflict" scenarios are the same author in two browser tabs, or the same account open on two devices, not simultaneous multi-user editing. Optimistic locking is the standard, idiomatic Rails mechanism for exactly this (no new gem, no new architecture), and directly satisfies FR-007 and the "two browser tabs" edge case by making a stale write fail loudly (server-detectable) instead of silently winning by request-arrival-order.

**Alternatives considered**:
- *Wall-clock timestamp comparison (`updated_at`) sent from client instead of a version counter*: rejected — clock skew between client and server, and the fact that `updated_at` is already touched by unrelated `after_save` hooks (`generate_snapshot`, `detect_locale_async`, both in `app/models/article.rb:117-120`), make a monotonic version counter more reliable than a timestamp for this purpose.
- *Full collaborative merge (OT/CRDT)*: rejected — far beyond this feature's scope (spec explicitly frames this as a single-author reliability fix, not a new collaboration feature); no product requirement calls for concurrent multi-author editing of one article.

## 4. What layout should replace the current two-tab (Edit / Options) structure, given full structural freedom (per Clarifications)?

**Decision**: Replace the tab switcher with a **persistent two-pane layout** on desktop: the writing surface (title, intro, rich-text body) as the primary, always-visible column, and a collapsible **Settings rail** docked to the side containing the grouped settings sections (see research item 6). On mobile, the Settings rail collapses into a bottom-sheet/slide-over triggered by an explicit "Settings" affordance in the top bar; the writing surface remains full-width and primary. A slim top bar hosts: logo/back, a save-status indicator (research item 5), a Focus Mode toggle (User Story 5), a Preview toggle (User Story 6), and the Publish action. Content and settings are simultaneously reachable — never mutually exclusive views — which structurally satisfies User Story 3 ("one unified editing experience": there's no longer a separate "which tab am I in" state to reason about) and User Story 2's requirement that settings are available immediately on a new, unsaved article (FR-015) without a tab switch.

**Rationale**: The current tab model is the direct cause of the "split save model" confusion (an author can be "in" the Edit tab and reasonably not realize Options changes need a separate save) — a persistent, always-visible settings surface removes that ambiguity by construction, independent of the autosave-unification work in research item 1. This is the single largest structural change of the feature and is deliberately chosen over lighter alternatives (see below) because the Clarifications session explicitly opted for full structural freedom, and a persistent side rail is the most common, well-understood pattern for "primary content + secondary configuration" editors (e.g., most modern CMS/blogging tools).

**Alternatives considered**:
- *Keep two-tab structure, just restyle*: rejected per Clarifications (`structural_freedom: fully_open`) — would leave the split-save perception problem unresolved even after autosave is technically unified.
- *Single long scrolling page (content, then settings, no separate surface at all)*: rejected — reintroduces the "settings are far away, easy to forget" problem the current design already has (just without tabs), and doesn't leave room for Focus Mode to cleanly hide "everything except the writing surface."
- *Modal/full-screen settings overlay*: rejected — modal overlays are appropriate for occasional, blocking actions (e.g., Publish confirmation, which already correctly uses one), not for a panel an author may want to glance at or adjust repeatedly while writing (price, tags) without losing their place in the document.

## 5. What are the concrete save-status states and how are they computed?

**Decision**: A client-side (Stimulus) state machine with four states — `idle` (no local changes since last confirmed save), `dirty` (local changes not yet sent, mid-debounce), `saving` (request in flight), `saved` (last request succeeded) — plus a distinct `error`/`offline` state (last request failed, retrying). Rendered as a small, unobtrusive, ARIA-live-region indicator in the top bar (and, per FR-025, still visible/accessible in Focus Mode). This directly replaces today's narrower `notSavedAlert` target (`article_form_controller.js:388-407`, `app/views/articles/_updated_at.html.erb`) which only ever signals "content autosave specifically failed," not the full state space FR-002 requires.

**Rationale**: All the necessary signals already exist client-side (the debounce timer, the in-flight request promise, the request's resolve/reject) — this is a state-machine refactor of existing instrumentation, not new infrastructure. An ARIA live region ensures the status change is announced to assistive technology, addressing accessibility for a persistent, no-explicit-save UI (an implicit requirement of removing a manual "Save" button — the author needs an *equally reliable*, not just visual, way to know their work landed).

**Alternatives considered**: A toast/notification per save — rejected, would be extremely noisy given saves happen every few seconds while typing; a persistent, low-key status indicator (as already partially exists via `_updated_at.html.erb`'s "last saved" timestamp) is the correct pattern and just needs to be generalized to cover settings saves and the full state space.

## 6. How should the settings panel be grouped, and how should the revenue split be simplified?

**Decision**: Reorganize the current flat, 12-field `_option_fields.html.erb` into five labeled sections: **Cover & Tags**, **Pricing & Access** (price, currency, free-content ratio), **Revenue Split** (guided plain-language summary by default; raw ratio fields for readers/author/references behind an explicit "Advanced" disclosure, platform/collection ratios always shown as read-only info, never inside the "Advanced" editable set), **References**, and **Collection**. The revenue-split summary is computed and re-rendered client-side (extending the ratio-calculation methods that already exist in `article_form_controller.js:212-294` — `updateReadersRevenueRatio`, `calReferenceRatio`, `calAuthorRevenueRatio` — with a `renderRevenueSummary()` that reads the same values already being computed) as plain-language text (e.g., "You keep 50% · Early readers 40% · Platform 10%"), with real-time inline validation reusing the same sum-to-1.0 rule already enforced server-side (`Article#ensure_revenue_ratios_sum_to_one`, `app/models/article.rb:415-417`) mirrored client-side so the author sees the problem before submitting, not just as a server error after the fact.

**Rationale**: The underlying math and bounds are already implemented and correct (readers/author/references auto-calculation already happens in JS today, per direct inspection); the problem the audit identified is purely presentational — six visually-identical fields with no distinction between "read-only/derived" and "author-editable," and no plain-language translation of what the numbers mean. Reusing the existing calculation functions (rather than rewriting the math) minimizes risk of introducing a new revenue-calculation bug in a monetized flow.

**Also fixed as part of this reorganization** (audit findings #5, #8 / FR-014, FR-016):
- The misplaced validation error at `app/views/articles/_option_fields.html.erb:67-69` (renders `form.object.errors.where(:intro)` beneath the *Collection* field) is corrected to show `:collection_id`/`:collection_revenue_ratio` errors there instead, with the real `:intro` errors surfaced next to the intro field in `_form.html.erb`.
- Disabled fields (currency, revenue ratios once published) gain a one-line explanatory caption (e.g., "Locked after publishing") rather than an unexplained `disabled` attribute.

**Alternatives considered**: A wizard-style multi-step settings flow (price → split → cover → references, one screen at a time) — rejected as overkill for a panel an author may want to glance at and adjust non-linearly (e.g., change price without re-walking a whole wizard); grouped sections in a single scrollable rail achieve the same clarity with less friction for repeat edits.

## 7. How should Publish Readiness (User Story 4) be computed?

**Decision**: Before rendering the existing publish-confirmation modal (`dashboard/published_articles#new`), run the article's existing validations in the `:publish` context (`@article.valid?` — which already runs `ensure_revenue_ratios_sum_to_one`, `ensure_references_ratios_correct`, `ensure_price_not_too_low`, and presence validations that are conditionally required `unless: :drafted?`) plus the existing `ensure_content_valid` AASM guard (title + content present), and surface `@article.errors.full_messages` directly in the confirmation modal as a specific, itemized list — instead of the current behavior where `may_publish?` (which only checks the AASM guard, not the fuller validation set) silently no-ops if false (`dashboard/published_articles_controller.rb:9-15`, `elsif @article.may_publish?` — if false, nothing happens and no message is shown to the author at all today).

**Rationale**: Reuses 100% of already-implemented, already-correct validation logic — no new business rules, just surfacing what the model already knows is wrong (or would be wrong) in a way the author can act on. Directly resolves FR-021/FR-022 and closes a real, confirmed gap: today, an author whose `may_publish?` guard fails (e.g., no title) simply sees nothing happen when they attempt to publish.

**Alternatives considered**: A separate `PublishReadinessService` object computing its own duplicate rule set — rejected as unnecessary; `Article`'s existing validations already are the single source of truth for "is this article valid," and reusing them via context-aware `valid?` avoids maintaining two parallel definitions of "ready to publish."

## 8. How should Live Reader Preview (User Story 6) actually render "exactly as a reader would see it," including the paywall boundary?

**Decision**: Replace the current dormant `articles#preview` (a `POST` action that just runs raw, unpersisted textarea content through `MarkdownRenderService.call(content, type: :full)` — with no paywall/free-content-ratio logic at all, confirmed unused anywhere in the current UI) with a `GET /articles/:uuid/preview` member action that reloads the **persisted** (autosaved) article and renders it using the exact same branching the public article page already uses: `app/views/articles/_content.html.erb` renders `articles/_full_content` when `article.authorized?(user)`, else `articles/_partial_content` (the paywall-fade + unlock-card partial, `app/views/articles/_partial_content.html.erb`). For the author's own preview, the branch is decided explicitly by `article.free?` (full content) vs. priced (always the partial/paywall view, since the point of the preview is to see what a reader *without* a purchase sees) — not by `authorized?(current_user)`, which would incorrectly show the author their own full content even for a priced article.

**Rationale**: Reuses the exact partials the public page already renders, guaranteeing pixel/typography parity (FR-027) by construction rather than by duplicating markup. Using the persisted/autosaved article (not raw in-progress textarea content) is both simpler (no need to re-implement `partial_content_as_html`'s truncation logic against arbitrary unsaved text) and consistent with the spec's own wording (FR-027: "current **autosaved** state") — and since autosave now lands within about a second of any edit (research item 1), this is barely distinguishable from a true live preview in practice.

**Alternatives considered**: Keep the existing POST-based raw-content preview and just also apply paywall logic to the posted string — rejected; would require re-implementing `partial_content_as_html`'s HTML-truncation algorithm against transient unsaved text, duplicating logic that already exists and is already tested against the persisted `content` column.

## 9. What should Focus Mode (User Story 5) hide, and how does it interact with the new persistent Settings rail?

**Decision**: A client-only (no server round-trip, no persistence) Stimulus-toggled view state that hides the top bar (collapsing to a minimal floating save-status pill) and collapses the Settings rail, leaving only the writing surface. Re-invoking the top bar (mouse-to-top-edge or a keyboard shortcut, e.g. `Esc`) restores the full chrome without altering any article data, scroll position, or cursor location — it is purely a CSS/visibility toggle on already-rendered DOM, not a re-render.

**Rationale**: Because Focus Mode touches no server state and no article data, it carries zero risk to the reliability work in User Stories 1–4; it can be implemented as a small, additive Stimulus behavior on top of the redesigned layout from research item 4, satisfying FR-025/FR-026 and the corresponding edge case (save-status warning must remain visible even in Focus Mode) by keeping the save-status pill outside whatever is being hidden.

**Alternatives considered**: Persisting the author's Focus Mode preference across sessions — explicitly not required by the spec (Key Entities describes Focus Mode as a "temporary, toggleable... display state"); adding persistence would be unrequested scope growth.

## 10. What should happen to the `dashboard/published_articles#update` stale-DOM-target bug (FR-024)?

**Decision**: Fix `app/views/dashboard/published_articles/update.turbo_stream.erb:10` — it currently targets `turbo_stream.replace "edit_article_#{@article.id}"`, but no element with that DOM id exists anywhere in the redesigned editor (the actual container id, per `app/views/articles/_edit_form.html.erb:4`, is `"#{dom_id article}_edit_form"`, i.e. `article_<uuid>_edit_form`). Correct the target to match, so a failed publish attempt's turbo_stream response actually re-renders the (now settings-visible, error-carrying) form instead of silently no-op'ing.

**Rationale**: Directly confirmed by reading both files; today a failed publish shows a warning flash (`t('article_not_completed')`) but the underlying form is never actually refreshed with the article's current (possibly now-annotated-with-errors) state, because the turbo_stream target doesn't match any real element. This is bundled with User Story 4 (Publish Readiness) since both are about making publish failures legible.

**Alternatives considered**: None — this is an unambiguous bug fix with one correct target id.
